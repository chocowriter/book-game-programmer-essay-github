Fix & Run 방식을 반복하다 보면 코드를 급하게 짜고 있다는 증거입니다. 느긋하게 코드를 짭니다. 그리고 예외까지 생각하면서 디버깅 할 시간을 단축합니다. 기능이 구현되었다고 해서 바로 Run 해서 버그 확인 해서는 안 됩니다. 두 세 시간 정도 하나의 기능에 집중하다가 버그도 없을 것이라는 확신이 들 때 Visual Studio에서 Run 버튼을 클릭해 봅니다. 그러면 예상한데로 기능 구현이 제대로 되었고 버그도 없는 깔끔한 상태가 됩니다. 저는 이럴 때 일종의 쾌감을 느낍니다. 어렵다고 느낀 기능 요구사항이 이렇게 Fix & Run 하지 않고 한 번에 동작하면 희열을 느끼고 뿌듯합니다.

그래서 제안합니다. Run 버튼을 클릭하지 않습니다. 코드를 수정하고 종이에 설계를 합니다. 예외 처리까지 꼼꼼하게 합니다. 테스트 코드까지 짤 수 있다면 그렇게 해봅니다. 하지만 현실은 녹록치 않습니다. 빨리 돌아가는 코드를 만들어야 한다는 압박감이 있습니다. 돌아가는 코드가 정답이라고 주장하는 개발자도 있습니다. 하지만 이런 경험을 해보았다면 느긋하게 코드를 짜는 것처럼 보이지만 디버깅으로 추가적인 시간이 들어가는 것을 방지할 수 있기 때문에 적극 권장합니다.
